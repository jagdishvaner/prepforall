Architecture Approach for Sub-project 1

Monorepo restructure + Frontend split + OVH K8s deployment — pick the approach that fits

A

Big Bang Migration

Restructure the entire repo in one go: add Turborepo, split frontend into marketing + platform apps, set up K8s manifests, CI/CD — then deploy everything together.

Pros

  • Clean break — no transitional state
  • Everything works together from day one
  • One PR to review

Cons

  • Large risky change — hard to debug if something breaks
  • Weeks before anything is deployable
  • Existing working code gets disrupted
B

Incremental Migration (Recommended)

Migrate in 4 ordered steps, each independently deployable and testable:

  1. Step 1: Add Turborepo + shared packages (ui, types, config). Existing apps/web keeps working.
  2. Step 2: Create apps/marketing (Next.js) — extract public/SEO pages from apps/web.
  3. Step 3: Create apps/platform (Vite + TanStack) — migrate authenticated pages.
  4. Step 4: Dockerize everything + K8s manifests + CI/CD for OVH.

Pros

  • Each step is small, testable, and reversible
  • Working software at every stage
  • Can ship Step 1-2 while Step 3-4 are in progress

Cons

  • Transitional period where old and new coexist
  • Slightly more total git history
C

Platform-First (Skip Marketing)

Build only apps/platform (Vite + TanStack) + K8s deployment. Skip the marketing site entirely for now — use a simple Notion page or Carrd for the landing page.

Pros

  • Fastest path to a working product
  • Marketing site can be built independently later
  • Focus all energy on the core platform

Cons

  • No SEO presence at launch
  • Marketing site retrofit later needs careful routing
  • Less professional appearance for university pitches
